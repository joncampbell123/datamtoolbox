After studying many different versions of DOS, the conclusion is this: The Standard MBR code was solidified with the release of IBM®/MS™ DOS 3.30, but its history actually begins with the release of IBM® Personal Computer™ DOS 2.00.

The only differences found to date in the MBRs created by all DOS 3.30 versions of the FDISK program are in a single version of DOS 3.30 made for NEC® Powermate Systems! (If you know of any other MBR code differences², please let me know.) The IBM® Personal Computer™ DOS version 3.30's MBR code is exactly the same code used in most versions of MS-DOS 3.30 (at least for those systems with an architecture close to an IBM® machine) and all other versions up to and including that used by the original Microsoft® Windows™ 95 FDISK.EXE program.

 

The Difference in Detail:

The different code for the NEC® MS-DOS 3.30 (Revision 3) MBR is shown below inside the YELLOW BOX in the following Table. (If you find any other versions of DOS that have the same kind of code; or variations of it, please inform me!)

The first four digits are the hex offset locations in the first Segment (0000:offset) of Memory where these instructions are executed during the boot-up process:

   7C18   JMP  0000:061D

   0620   MOV  BL,04

   061D   MOV  SI,07BE
   0620   MOV  BL,04
 
 * 0622   MOV  DI,077C
 * 0625   CMP  WORD PTR [DI],A55A
 * 0629   JNZ  0630
 * 062B   MOV  BL,08
 * 062D   MOV  SI,077E

   0630   CMP  BYTE PTR [SI],80

Note how both MBRs have the same bytes in yellow after those which were added, but at different offset locations!

* The code here checks for the existence of the Hex Word A55Ah at location 077C in Memory. If it doesn't exist, then execution simply jumps to location 0630 and continues as it would for the Standard MS-DOS 3.30 code. However, if it is found, then the values previously placed into the BL and SI registers (04 and 07BE) are replaced with the alternate values of 08 and 077E instead. This clearly implies the existence of a Partition Table with eight entries beginning at location 077E (rather than the usual four entries which begin at 07BE). These entries must be set up and placed there by the same (FDISK) program which sets the bytes at locations 077C and 077D to 5A and A5 giving a Table that looks like this in the hard drive's MBR sector:

Well, I had to test this out as soon as possible, so I ran it under a Windows version of the BOCHS x86 Emulator. And although I wasn't at all sure how large of a partition it could create, the evidence for an 8-entry Partition Table with 8 Primary DOS partitions was irrefutable!

When I added partitions to the Table, the first one was placed at offsets 07EEh through 07FDh and the next one just above it. So, the entries are counted backwards from that of a normal Table. Thus, looking at such a Table with a disk editor or simple MBR utility, it would show the first NEC eight-entry partition as being the last one (#4) in a normal Table.

One interesting thing about this, is the fact that these special NEC entries had their own Partition-Type identifier (a 24h byte; known as "NEC DOS 3.x") so any other OS, booting up the drive wouldn't mistake it as a normal 32MB DOS partition. [For more info on Partition Types, see Andries Brouwer's page: http://www.win.tue.nl/~aeb/partitions/partition_types.html.]

For compatibility with other DOS systems, the FDISK program was made so it could also create a standard 4-entry Partition Table using its DOS 3.3 Compatibility mode.

As best I can figure out, it appears that the FDISK.COM program with this version of DOS can partition a hard drive into a single partition up to somewhere higher than 443 Binary Mega Bytes (900 cylinders x 16 heads x 63 sectors); at least as far as using the BOCHS Emulator that is, with which I had many problems³ attempting to complete this task!

Furthermore, when I finally made the effort to connect a real hard drive (of 540 MB) to another machine, this NEC® DOS 3.30 seemed to get confused at times about how to partition the drive, and couldn't create any partitions over 256 Binary MB (522 cylinders x 16 heads x 63 sectors)! This was a very frustrating time for the author! It appears that the FDISK and other programs are somehow hard-coded to require a drive with less than 1024 cylinders (if even that) for getting the best capacity usage, and/or some type of BIOS chip that it understands! Under BOCHS, I got this error message whenever the diskette was first boot up (the second line is a 'typo error' built into the program, and is exactly how it appeared on screen):

 Discrepancies have been found in the information
 stored the in permanent memory of this computer.
You must use the Hardware System Setup to correct
this information.

Press any key to continue.

Forgetting about all the partition-size problems I had, one main point is this:

At first I had wondered how any DOS 3.30 version (given the limitations I knew it had) could create a partition greater than 32MB. After checking the dates for FDISK.COM, FORMAT.COM and other files on the diskette and comparing them with those from later versions of DOS, it became apparent that this diskette's abilities were highly likely to have been based upon programming from IBM®'s DOS 4.00 which had been finalized in June of 1988 and/or the Microsoft® MS-DOS 4.00 which came out later in October that same year. Thus the date for the FDISK program on this NEC diskette of August 8, 1988 is around the same time that Microsoft had at least acquired the ability to handle partitions much greater in size than just 32MB. In any case, there are five files on this diskette in a "SETUP" directory which are all dated "9-18-89 11:41 am" so it couldn't have been released by NEC® until at least 4 months after MS-DOS 4.01 (released April 7, 1989) was already in existence. Much more surprising, is COMPAQ® MS™-DOS 3.31 (released in November of 1989) which could support multiple partitions of 512 MB and was available only about a month and a half after the NEC® DOS.

I was able to locate and test a "G" version (released in 1990) of COMPAQ®'s Personal Computer DOS 3.31, and it uses a partition type of 06h which is the normal FAT16 identifier (for partitions greater than 32 MB), and it did create a single partition of about 512 Binary MB as stated.

 

Summary:

Although I found many FDISK.COM (and .EXE) programs with different MD5 signatures, the MBR code that they all create is identical to the Standard MS-DOS 3.30 version of FDISK (with the NEC version¹ being the only exception I've found to date). The MD5 sums, file dates and sizes for all the copies of FDISK which have been examined are listed here.

After going back even further and examining the code used in the MBRs for IBM® Personal Computer™ DOS versions 3.21, 3.20, 3.10, 3.00 and even version 2.00, I found only some minor variations between them and the code used by DOS 3.30; mostly dealing with how the error messages were displayed. [The details of these differences are explained on my IBM® Personal Computer™ DOS 2.00 MBR page.]

Since what we're calling "the Standard MBR code" here at The Starman's Realm is still being used by a number of Partition Table and MBR utilities and data recovery tools (such as Test Disk), that means its history has already spanned two decades of hard drive innovations! So, this study has certainly brought about some justification for the phrase Standard MBR code (which is rather fitting since my first endeavor here was titled The Standard MBR; to understand and then share with others how an MBR was involved in booting up a computer's operating system from a hard drive)!

NEC MS-DOS 3.30 FDISK.COM  55,029 bytes  August 8, 1988 20:39.00

It should be noted that I was not allowed access to the original diskette; only to a copy that did not have its write-protect notch covered; I know that for sure, because the Boot Record had been 'compromised' by a Windows OS at some point having its original OEM ID replaced by an "IHC" string!

However, all the files in the Directory were in alphabetical-order, and it did not contain any deleted filenames. So, I assume it was the original FDISK.COM file which I examined. If you can corroborate the data on this page from your own collection or printed materials, please let me know.

² Apart from DOS versions which are very different that is, such as Zenith Data System's MS-DOS "R" version which uses a PART.EXE program to set up drives (there's no FDISK in this version), and doesn't even appear to have a Partition Table at first glance!

³ At first, when I used an image that had more than 1024 cylinders, this FDISK program said there were only 514 cylinders available, and that's what it said it would use to partition it (I didn't try to actually carry out the formatting of the image at that time). Later on, I thought that was a bit weird and tried again with an image that was exactly 1024 cylinders. Well, that time it recognized all of them, and said it could make a partition of 0 to 1022 cylinders (apparently reserving 1 at the end for a "Test Cylinder"; a common practice back then). But when the diskette was boot up again to attempt formatting after it set the partition Active, it locked-up the BOCHS Emulator every time! Apparently either Bochs and/or this DOS running in an emulator didn't like the image it set up for some reason... BUT by trial and error, I finally found that I could come up with a partition entry of 900 cylinders (and it would actually carry out the formatting process!), but using one much larger than that caused various errors in BOCHS; for example, when I tried 924 cylinders, I got:

>>PANIC<<  prefetch: RIP > CS.limit

So, my conclusion at this time is that the NEC® DOS 3.30 (Rev. 3) diskette is able to at least work on a hard drive image of about 900 cylinders (I can only guess it would work on a real drive of this size since I do not have one; and would actually like to use various sizes close to this!) AND ALSO that it will for sure have problems with a real drive of 1046 cylinders (with 16 heads and 63 sectors). If anyone has come across an article, e-mail, or any other literature that describes the limits of this particular version of NEC® MS-DOS, please let me know!
